# Provenance Guard

A platform for sharing original creative writing that helps readers judge whether a
submitted piece was **written by a human** or **generated by AI and misrepresented as
human work** — and communicates that judgment *honestly*, including how uncertain it is.

Provenance Guard never returns a bare verdict. It runs two independent detectors, fuses
them into a calibrated probability, and turns that into a plain-language transparency
label whose wording reflects the confidence. Every decision is rate-limited, persisted,
and written to a structured audit log, and creators can contest any classification.

> The full design rationale lives in [planning.md](planning.md). This README is the
> operator/portfolio view: how it works, why, and where it breaks.

---

## Running it

```bash
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env            # then set GROQ_API_KEY in .env
python app.py                   # serves http://127.0.0.1:5000
```

Submit some writing (the included client keeps the shell out of JSON construction):

```bash
python submit.py --creator alice --text "I've been thinking — really thinking."
python submit.py --creator alice --file story.txt
```

| Method & path | Purpose |
|---|---|
| `POST /submit` | Classify a submission (`text`, `creator_id`). Rate-limited. |
| `GET /log?limit=N` | Most recent audit entries, newest first. |
| `POST /appeal` | Contest a classification (`content_id`, `creator_reasoning`). |
| `GET /` | Health check. |

> The LLM signal needs a Groq API key. Without one the system still runs and degrades
> to a single-signal, low-reliability result (see [Spec reflection](#spec-reflection)).

## Architecture overview

The path a submission takes from input to transparency label:

```
POST /submit
   │
   ▼
[Rate Limiter]  ── over limit ─▶ 429 (no pipeline work, no LLM spend)
   │
   ▼
[Validator]     ── bad body ───▶ 400
   │  text + creator_id
   ▼
[Content Repository]  assign content_id (UUID), status = "classified"
   │
   ▼
[Detection Orchestrator]  ── fans out to two independent signals ──┐
   │                                                               │
   ├─▶ Signal 1: Stylometric Analyzer  (local, deterministic)      │
   ├─▶ Signal 2: LLM Detector (Groq)   (semantic)                  │
   │                                                               │
   ▼                                                               │
[Confidence Aggregator]  reliability·trust-weighted mean;  ◀───────┘
   │  disagreement pulls toward 0.5  →  fused p_ai
   ▼
[Transparency Label Builder]  p_ai band → reader label
   │
   ▼
[Audit Logger]  structured JSONL record (signals, scores, label, model)
   │
   ▼
JSON response: content_id, creator_id, timestamp, attribution, confidence, llm_score, status
```

A separate `POST /appeal` path revisits an existing decision: it flips the content's
status to `under_review` in the repository and appends an `appeal` event to the audit
log alongside the original decision. No automated reclassification occurs.

**Code map.** [app.py](app.py) (HTTP + rate limiting) · [pipeline.py](pipeline.py)
(orchestrator) · [stylometry.py](stylometry.py) (Signal 1) · [llm_detector.py](llm_detector.py)
(Signal 2) · [aggregator.py](aggregator.py) (fusion) · [labels.py](labels.py) (labels) ·
[store.py](store.py) (content repository) · [audit.py](audit.py) (audit log) ·
[submit.py](submit.py) (client).

## Detection signals

Two signals of **different kinds** so each covers the other's blind spot.

### Signal 1 — Stylometric Analyzer ([stylometry.py](stylometry.py))

Pure-Python, no network, deterministic. It measures four classical features and maps
each to an "AI-likeness" sub-score, then takes a weighted blend:

| Feature | What it measures | AI direction | Weight |
|---|---|---|---|
| Burstiness (CV) | global variation in sentence length | low variation → AI | 0.35 |
| Sentence-length regularity (MASD) | local sentence-to-sentence rhythm | smooth → AI | 0.25 |
| N-gram repetition (trigrams) | repeated word sequences | more repeats → AI | 0.20 |
| Lexical diversity (MATTR) | length-robust vocabulary richness | lower → AI | 0.20 |

**Why chosen.** Human prose is *bursty and uneven*; AI text tends to be smoother and
more uniform. Stylometry is free, instant, offline, and fully reproducible, so it works
even when the LLM is unavailable and never leaks text to a third party.

**What it misses.** It is a weak *discriminator*: well-edited AI with deliberately varied
sentence lengths reads as "human," and — conversely — it *false-flags* formal human
craft built on repetition (villanelles, refrains, nursery rhymes) as AI. It also needs
enough text; on one or two sentences its variation metrics are noise, which the
`reliability` score reflects (it goes to near-zero). Because of this, the aggregator
trusts it *less* than the LLM (see below).

### Signal 2 — LLM Detector ([llm_detector.py](llm_detector.py))

A Groq-hosted model (default `llama-3.3-70b-versatile`) is prompted as a forensic
linguist and returns JSON `{p_ai, rationale}`. We **calibrate** its output: clamp away
impossible certainty and shrink 10% toward 0.5, because LLM self-reported probabilities
are overconfident.

**Why chosen.** It reads *semantics* — voice, idiosyncrasy, specificity, emotional
texture — exactly the dimensions stylometry is blind to. It is the stronger
discriminator, so it carries more weight in fusion.

**What it misses.** It is nondeterministic (the same text can score differently run to
run), it can be *confidently wrong*, it is **evadable** by "humanizer"/paraphrase tools
built to defeat detectors, and it costs money and latency per call. If the call fails or
no key is set, it returns `reliability: 0.0` and is ignored rather than crashing the
request.

## Confidence scoring

### How the signals are combined ([aggregator.py](aggregator.py))

Each signal reports `p_ai` (probability the text is AI-generated) and a `reliability`.
Fusion is a **reliability × trust-prior weighted mean**, then a **disagreement shrink**:

1. **Weight** = `reliability × trust_prior`, where trust priors are config:
   `stylometry = 0.5`, `llm = 1.0`. Stylometry is down-weighted because it is the weaker
   discriminator — this stops a confidently-wrong statistical signal from outvoting a
   reliable semantic one.
2. **Weighted mean** `m` of the signals' `p_ai`.
3. **Disagreement** `d = min(1, 2·weighted_std)` ∈ [0, 1].
4. **Fused score** `= 0.5 + (m − 0.5)·(1 − d)^0.5`. Agreement preserves the mean; conflict
   pulls toward 0.5 (uncertain). The `sqrt` makes the shrink *gentle* — a confident,
   reliable verdict isn't flattened to 0.5 just because one weak signal disagrees.

If only one signal is usable (e.g. no Groq key), the system falls back to that signal's
score directly. `confidence` in the API is this fused `p_ai`; **certainty is its distance
from 0.5**, which the label makes legible.

### How it was validated as meaningful

The fusion was checked to ensure agreement stays confident while conflict reads as
uncertainty (actual `aggregate()` outputs):

| Scenario (styl / llm p_ai) | Fused | Disagreement | Reads as |
|---|---|---|---|
| Agree, high (0.85 / 0.80) | **0.808** | 0.05 | confident AI |
| Agree, low (0.15 / 0.20) | **0.192** | 0.05 | confident human |
| Disagree (0.85 / 0.15) | **0.416** | 0.64 | pulled to uncertain |
| LLM offline (styl only) | **0.850** | 0.00 | single-signal fallback |

It also varies meaningfully across real inputs — see the two worked examples below
(0.800 vs 0.452).

### Two example submissions (actual scores)

> Stylometry values and the fusion arithmetic are computed by the code and reproducible.
> The LLM `p_ai` shown is a representative model response; live values vary slightly per
> call.

**Example A — higher confidence.** A uniform, generic six-sentence paragraph
("*The platform delivers consistent value to every user… It helps organizations achieve
their goals in a scalable and effective manner.*"):

| Signal | p_ai | reliability |
|---|---|---|
| stylometry | 0.618 | 0.437 |
| llm | 0.930 | 0.652 |
| **fused** | **0.800** (disagreement 0.27) | |

→ **"Probably AI-generated"** — both signals lean AI; the score sits clearly above the
midpoint.

**Example B — lower confidence.** A short, ambiguous single line
("*It was a bright cold day in April, and the clocks were striking thirteen.*"):

| Signal | p_ai | reliability |
|---|---|---|
| stylometry | 0.384 | 0.110 (one sentence → thin evidence) |
| llm | 0.470 | 0.184 |
| **fused** | **0.452** (disagreement 0.07) | |

→ **"Inconclusive — we can't reliably tell"** — little text and no signal far from 0.5,
so the system says so instead of guessing.

The gap (0.800 vs 0.452) crosses two distinct label categories — exactly the behavior the
spec asks for (0.51 and 0.95 must read differently).

## Transparency label

The label is driven by the **confidence band, not the raw float** ([labels.py](labels.py)).
Thresholds are configuration (overridable via env vars such as `PG_THRESHOLD_LEANS_AI`),
not magic numbers, and the reader copy is kept separate so either can be tuned.

| `p_ai` band | attribution | label (exact text) | gloss (exact text) |
|---|---|---|---|
| `< 0.15` | `human` | **Very likely human-written** | Our checks strongly point to a human author. |
| `< 0.35` | `leans_human` | **Probably human-written** | Our checks lean toward a human author, with some uncertainty. |
| `< 0.65` | `uncertain` | **Inconclusive — we can't reliably tell** | Our checks disagree or are inconclusive; treat authorship as unverified. |
| `< 0.85` | `leans_ai` | **Probably AI-generated** | Our checks lean toward AI generation, with some uncertainty. |
| `≥ 0.85` | `ai` | **Very likely AI-generated** | Our checks strongly point to AI generation. |

### The three required variants, written out

**High-confidence AI** (`p_ai ≥ 0.85`)
```
Very likely AI-generated
Our checks strongly point to AI generation.
```

**Human** (`p_ai < 0.15`)
```
Very likely human-written
Our checks strongly point to a human author.
```

**Uncertain** (`0.35 ≤ p_ai < 0.65`)
```
Inconclusive — we can't reliably tell
Our checks disagree or are inconclusive; treat authorship as unverified.
```

(The two intermediate bands — *Probably human-written* and *Probably AI-generated* — give
readers a calibrated middle ground rather than forcing a binary.)

### Typed description — leans AI, leans human, uncertain

The exact transparency-label fields the API returns for each of these three variants. The
`label` and `label_detail` text is fixed by the band; `attribution` is the band's type.

**Leans AI** (`0.65 ≤ p_ai < 0.85`)

Example submission:

> V-JEPA 2s paper reports pretraining a joint-embedding predictive architecture on over 1 million hours of internet video, then applying it to motion understanding, video question answering, and robot planning via a latent action-conditioned world model.

```json
{
  "attribution": "leans_ai",
  "label": "Probably AI-generated",
  "label_detail": "Our checks lean toward AI generation, with some uncertainty."
}
```

**Leans human** (`0.15 ≤ p_ai < 0.35`)

Example submission:

> I graduated with a Bachelors degree in CS with a good GPA, but I dont like coding at all. I cant even do a leetcode medium without external help and I dont want to watch tutorials and read books about it because I hate it even more. I dont want to learn patterns of coding problems because it makes me extremely upset and anxious.

```json
{
  "attribution": "leans_human",
  "label": "Probably human-written",
  "label_detail": "Our checks lean toward a human author, with some uncertainty."
}
```

**Uncertain** (`0.35 ≤ p_ai < 0.65`)

Example submission:

> At the World Cup, Iran drew three times, and had three goals called off-side. The most heart-breaking was the last, which would have secured a victory over Egypt. Despite their third-place group finish, Iran could have made the play-offs for the first time, if any of the three scenarios had occurred.
```json
{
  "attribution": "uncertain",
  "label": "Inconclusive — we can't reliably tell",
  "label_detail": "Our checks disagree or are inconclusive; treat authorship as unverified."
}
```

## Rate limiting

`POST /submit` is limited per client IP ([app.py](app.py)) using Flask-Limiter with
in-memory storage:

```
10 per minute; 60 per hour; 200 per day
```

Each submission triggers a paid LLM call, so these limits double as **cost control**.
Reasoning for the specific values:

- **10 / minute** — burst headroom for a writer revising and resubmitting, plus the odd
  retry after a 400/429. A human can't meaningfully produce 10 distinct pieces a minute;
  a flooding script is cut off within seconds.
- **60 / hour** — holds a *sustained* rate to ~1 submission/minute, already very active
  for creative writing, so nobody can ride the per-minute limit indefinitely.
- **200 / day** — an overall ceiling well above any genuine single-author daily volume,
  capping slow/distributed abuse and total LLM spend.

Over-limit requests return a JSON `429` **before** the handler runs — verified: in a
13-request burst, 10 succeed, 3 are rejected, and exactly 10 LLM calls are made.

## Known limitations

**Formal, repetition-driven poetry → false "AI" (demonstrated).** A villanelle, a
refrain-based poem, or a nursery-rhyme-style piece has low sentence-length variation,
heavy repetition, and small vocabulary — precisely the features stylometry treats as
"machine-smooth." Running the analyzer on the repeated refrain of *Do not go gentle into
that good night* yields a stylometry `p_ai` of **~0.91**: it flags skilled human craft as AI.
The repetition *is* the art, not an artifact of generation. The LLM signal can partly
rescue these cases, but with the LLM offline this is a confident false positive, and the
short length keeps reliability low.

Two further weak spots: **adversarially "humanized" AI** (text run through a paraphraser
to inject burstiness) is the dangerous *false negative* — it's built to beat stylometry
and leans entirely on the evadable LLM signal; and **very short submissions** (flash
fiction, micro-poetry) carry so little statistical evidence that the honest output is
"Inconclusive" with low reliability rather than a real verdict.

## Spec reflection

**One way the spec helped.** The requirement that confidence *reflect uncertainty* — "a
level of 0.51 should result in a distinctly different label than 0.95" — turned a vague
"return a score" into a concrete design driver. It produced the band-based labels and,
more importantly, the **disagreement-widens-uncertainty** fusion: without that pressure I
might have shipped a simple weighted average that reports a falsely confident number when
the two signals flatly contradict each other.

**One way implementation diverged, and why.** The spec states single-signal detection "is
not acceptable." The happy path honors this (stylometry + LLM), but the **failure path
diverges**: if Groq is unavailable or unkeyed, the system degrades to a single-signal,
stylometry-only result rather than refusing the request. The reasoning is availability —
returning a clearly-labeled, low-reliability estimate is better operator behavior than a
500, and the low reliability + "Inconclusive"-leaning output keeps it honest about the
weaker evidence. (Relatedly, the simplified `/submit` response no longer returns the
`label` itself; it is still computed and persisted to the store and audit log for the
reader-facing view, just not in the creator's submit acknowledgment.)

## AI usage

This project was built with an AI assistant under direction; the substantive design
calls and corrections were made by me. Specific instances:

1. **Confidence aggregator — overrode the fusion math.** I directed the AI to build the
   component that combines the two signals. Its first version took a plain
   reliability-weighted mean and pulled the result toward 0.5 *linearly* with
   disagreement. I found that this over-penalized the most useful case — a reliable
   signal standing against a weak, wrong one — and could rank a clearly-AI text below a
   genuinely ambiguous one. I overrode it with **per-signal trust priors** (stylometry
   weighted at 0.5 vs the LLM at 1.0, since stylometry is the weaker discriminator) and a
   **gentler `sqrt` disagreement shrink**, both exposed as config rather than constants.

2. **Stylometric analyzer — revised tokenization.** I directed the AI to write the
   four-feature analyzer. Testing on real prose, I caught that smart quotes from
   Word/Google Docs (U+2019) broke tokenization — `don't` split into `don` + `t`,
   corrupting token counts, lexical diversity, and repetition for essentially all
   real-world text. I added a typographic-normalization step before tokenizing, which the
   AI's version lacked.

3. **Submission UX — added a client and tightened error handling.** Hand-testing the
   endpoint produced confusing failures that traced to the shell mangling JSON
   (an apostrophe ending a single-quoted `curl -d '...'`). Rather than accept "it never
   reached the server," I added a stdlib [submit.py](submit.py) client that builds JSON
   correctly, and revised the `400` path to distinguish *unparseable JSON* from a
   *well-formed non-object* with an actionable message.

(Other directed-then-reviewed work: requiring label thresholds to be config not magic
numbers, and specifying the exact field order of the `/submit` response — which exposed
that Flask 3 sorts JSON keys by default, fixed with `app.json.sort_keys = False`.)
